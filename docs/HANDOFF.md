## 1. Контекст

Модуль «Логистика» AI-First ERP. Два плеча:
- **Доставка клиентам по РБ/РФ** через перевозчиков (DPD, Автолайт Экспресс, СДЭК, Европочта, Белпочта).
- **Импорт из Китая** (фабрика → консолидация → в пути → таможня → приёмка на склад).

Валюта BYN. Стек: фронт Next.js 15 / React 19 / TS / Tailwind 3.4 / App Router; бэк FastAPI / Python (`aios`, `core/`, `modules/<name>/`, Alembic, CI gate 90%).

**Жёсткие принципы проекта (соблюдать в любом коде):**
- AI-функции = **Итерация 1**. В прототипе нулевой функциональный AI; AI-элементы — ghost-плейсхолдеры (фиолетовый пунктир, бейдж «Итерация 1 · ИИ»). Бэкенд оставляет поле `insight` пустым, фронт его бейджит.
- **Человек всегда на карточке** — у каждой карточки ответственный (Диспетчер/Логист), даже если есть AI-агент.
- **Gross profit, а не объём** — KPI вокруг маржи (расход на доставку как % валовой прибыли).
- **Разделение ролей** — кладовщик/сборщик/диспетчер выполняют физ. операции; коммерческие действия (цена, резерв) принадлежат CRM/продажам.
- **Дизайн-система строгая** — Golos Text, сетка `236px / 1fr / 320px`, канбан-доминантный layout + правый рейл (чаты + AI-инбокс), единая anatomy карточек (ID, бейдж приоритета, ответственный, AI-бейдж, кнопки).

## 2. Состояние БЭКЕНДА (`LOG-6`, ветка `main`) — РЕАЛИЗОВАНО и ЗАПУШЕНО

`LOG-6` — это submodule `modules/logistics`. Файлы лежат **в корне** репозитория `LOG-6` (не под `modules/logistics/`).

Файлы и что в них:
- **`stages.py`** — стадии воронок.
  - `DELIVERY_STAGES`: `planned`(Заявка/#3B82F6) → `assigned`(Перевозчик назначен/#8B5CF6) → `in_transit`(В пути/#F59E0B) → `delivered`(Доставлено/#22C55E).
  - `IMPORT_STAGES`: `factory` → `consolidation` → `in_transit` → `customs`(#EC4899) → `warehouse`(#22C55E).
- **`models.py`** — `Shipment`, `ImportShipment`, `Carrier` (schema `logistics.*`, у всех новых колонок `server_default`).
  - `Shipment`: + `carrier_code`, `carrier_order_no`, `payer` ("компания"/"клиент"), `tracking_status`.
  - `Carrier`: + `code`, `track_url`.
- **`schemas.py`** — Pydantic: `CarrierCatalogItem`, `CarrierOrderRequest`, `TrackingUpdate`, `CarrierCostStat`, `CostReportOut`, `DashboardOut` (+`shipping_cost_company`, `cost_by_carrier`). Константа `CARRIERS_RB` (5 перевозчиков РБ с шаблонами `track_url`).
- **`routes.py`** — prefix `/logistics`, 16 эндпоинтов. Ключевые:
  - `POST /shipments/{id}/carrier-order` — назначает перевозчика, генерит номер заказа, ставит стоимость, `planned→assigned`, эмитит `logistics.carrier_order.created`.
  - `PATCH /shipments/{id}/tracking` — обновляет трекинг-статус.
  - `GET /carriers/catalog`, `POST /carriers/seed` (идемпотентно по `code`).
  - `GET /costs` — total/company/client/import_cost/by_carrier (клиентские исключены из расхода компании; `CLIENT_PAYER="клиент"`).
- **`module.py`** (v0.2.0) — `include_router`, подписка `sales.document.posted` → `on_document_posted`, 2 виджета (Логистика, Импорт из Китая).
- **`events.py`** — НЕ менялся (совместим): на order-kind создаёт `Shipment`, эмитит `logistics.shipment.created`.

## 3. Состояние ФРОНТЕНДА — эталонный HTML-макет `ui/logistics-module.html`

Self-contained HTML (всё inline, Golos Text через Google Fonts). **Это дизайн-референс**, не продакшн-код. По нему реализуем React-модуль (паттерн как `leads-workspace.tsx`).

**6 вкладок:**
1. **Доставка РБ/РФ** — канбан 4 стадии (Заявка → Перевозчик назначен → В пути → Доставлено). KPI: Грузы в пути, Просрочки SLA, **OTIF**, **FADR (с 1-й попытки)**, Расход компании (% валовой прибыли). Карточки: объёмный вес (мест/физ./об. → оплачиваемый), доля доставки в марже, COD-чип, документ ТТН-1, POD (получатель+время+подпись), самовывоз, бейдж «Просрочено», чип «Клиент уведомлён».
2. **Импорт из Китая** — канбан 5 стадий (флаг, контейнеры, Incoterms, PO, себестоимость).
3. **Перевозчики → Scorecard** — взвешенная оценка A/B/C: в срок, брак-free %, точность счетов %, претензии %, ставка/дост.; формула (в срок 40% · брак 25% · счета 20% · претензии 15%); балл управляет распределением объёмов.
4. **Тарифы и зоны** (справочник, как «Нормы» в Производстве) — таблица зон (4 зоны РБ) + прайс-матрица (перевозчик × зона × весовая ступень + забор/наложка/страховка). Источник для расчёта в окне оформления.
5. **Проблемы и возвраты** — канбан исключений: недозвон → отказ/повреждение → переадресация → возврат на склад (эскалация, претензия→Юрист, передача в Склад).
6. **Расходы** — дашборд: доставка к валовой прибыли (4.2%), COD (к возврату/возвращено/комиссия), расход по перевозчикам, **аудит счетов** (факт vs тариф из справочника, расхождения).

**Правый рейл:** Чаты + ИИ-координатор (инбокс-превью с Принять/Назначить/Отклонить — задизейблено, Итерация 1). Тумблер «Режим ИИ» в шапке.

**Применённые best-practices (из ресёрча, отражены в макете):**
OTIF (в срок и в полном), FADR (лидеры >93%, провал <3%), Perfect Order (~84% норма), Scorecard перевозчика, аудит счетов (5–6% счетов с ошибкой), уведомления клиента (рычаг FADR), объёмный вес, расход к выручке/марже, POD, обработка исключений, консолидация.

## 6. ИДЕЯ — Личный кабинет клиента (Customer Portal)

> Статус: **идея / бэклог**, не реализовано. Зафиксировано 2026-06-11.
> Внешний (B2B) кабинет для контрагента-покупателя поверх данных логистики/sales/wms.

### Зачем
Снять нагрузку с диспетчера/менеджера по типовым обращениям («где мой груз», «что есть на
складе», «пришлите акт») и дать клиенту self-service. Прямой рычаг **FADR/уведомлений**
(см. §3): информированный клиент = меньше провалов доставки и претензий.

### Что видит/делает клиент (5 блоков)
1. **Мои покупки** — история заказов клиента (из sales): № заказа, дата, состав, сумма (BYN),
   статус оплаты/проводки. Read-only витрина по `counterparty`.
2. **Отслеживание груза** — его отгрузки (`logistics.shipment` / `import_shipment`) с воронкой
   стадий (Заявка → Перевозчик назначен → В пути → Доставлено) и таймлайном `tracking_status`,
   ETA, ссылкой `track_url` перевозчика, POD после доставки. Только грузы этого клиента.
3. **Наличие на складе** — витрина остатков (из wms / `core.stock`): что доступно к заказу,
   количество (или чип «в наличии / под заказ» без точных цифр, если так решит политика),
   цена из прайса. **Без** внутренних себестоимостей и резервов.
4. **Заявка на покупку** — клиент формирует заявку (корзина из позиций склада) → создаётся
   «запрос», который падает менеджеру sales в обычную воронку (НЕ автозаказ). Коммерческое
   действие (цена/резерв/подтверждение) остаётся за продажами — принцип разделения ролей (§1).
5. **Документы** — просмотр и скачивание привязанных к клиенту документов: ТТН-1, счета,
   акты, POD. Только свои документы.

### Уровни доступа клиента (access tiers) — настраиваются менеджером per-контрагент
Не все клиенты равны: «рискованным» даём минимум, доверенным — больше данных по товарам и
ценам. Уровень хранится на контрагенте (`portal_access_level`), назначается менеджером sales,
управляет тем, **что и в каком объёме** видно в портале. Эскиз шкалы (уточнить при реализации):

| Уровень | Кому | Покупки | Трекинг | Склад (наличие) | Цены | Заявки | Документы |
|---------|------|---------|---------|-----------------|------|--------|-----------|
| **`restricted`** (мин.) | новый / проблемный / с долгом | только свои заказы | базовый статус | ❌ скрыт | ❌ | ❌ (только связь с менеджером) | только ТТН/POD по своим грузам |
| **`standard`** (база) | обычный клиент | ✅ | ✅ полный таймлайн+ETA | чип «в наличии / под заказ» (без цифр) | прайс по запросу | ✅ заявка → менеджеру | счета/акты/ТТН/POD |
| **`trusted`** (расширенный) | проверенный, регулярный | ✅ + аналитика | ✅ | точные остатки (qty) | персональный прайс/скидка | ✅ + быстрый дозаказ | + закрывающие, сверки |
| **`partner`** (макс.) | дилер / постоянный B2B | ✅ + история по всем точкам | ✅ | остатки + резервирование | контрактные цены, спецусловия | ✅ заявка может авто-создавать черновик заказа | полный пакет + выгрузка реестром |

Принципы:
- Уровень — **белый список возможностей**, бэкенд проверяет его на каждом запросе (не доверять фронту).
  «Рискованному» физически не отдавать остатки/цены, даже если он знает URL эндпоинта.
- **Раскрытие цен и остатков** — два независимых флага поверх уровня (можно дать `standard`+видит цены,
  но без точных остатков, и наоборот) → гибче, чем жёсткая лестница.
- Смена уровня — действие менеджера sales (коммерческое решение), логируется в аудит.
- Дефолт для нового контрагента — **`restricted`**, повышение вручную.

### Данные / контракт (эскиз, уточнить при реализации)
- **Привязка** — всё фильтруется по `counterparty_id` (контрагент из shared kernel `core.domain`).
- **Уровень доступа** — поле `portal_access_level` (`restricted`/`standard`/`trusted`/`partner`,
  default `restricted`) + 2 флага `show_prices`, `show_stock_qty` на контрагенте/профиле портала.
  Каждый `/portal/*` эндпоинт фильтрует ответ по уровню (server-side gate, см. таблицу выше).
- Новых таблиц логистики, скорее всего, не нужно для блоков 1–3, 5 (читаем существующее
  sales/logistics/wms через `core`/события — модули не лезут в чужие схемы, §2.4).
- **`portal_purchase_request`** (новая, схема `logistics` или отдельный портал-модуль) — заявка
  клиента: `id`, `counterparty_id`, `items` (jsonb: sku, qty), `comment`, `status`
  (`new`→`accepted`→`rejected`), `created_at`, `created_by` (контакт клиента). Эмит события
  `portal.purchase_request.created` → подписывается sales (создаёт лид/заказ-черновик).
- **Документы** — реестр ссылок (id, тип, `counterparty_id`, `entity_ref`, путь/URL хранилища,
  выдача по signed-URL). Скачивание — через бэкенд с проверкой принадлежности клиенту.

### Эндпоинты (эскиз, отдельный префикс `/portal`, аутентификация клиента ≠ внутренний Keycloak-стафф)
- `GET /portal/orders` — мои покупки.
- `GET /portal/shipments` + `GET /portal/shipments/{id}/tracking` — мои грузы и трекинг.
- `GET /portal/stock` — витрина остатков (политика раскрытия цифр — флагом).
- `POST /portal/purchase-requests` — отправить заявку; `GET /portal/purchase-requests` — мои заявки.
- `GET /portal/documents` + `GET /portal/documents/{id}/download` — список и скачивание (signed-URL).

### UI (как вкладка-референс или отдельный мини-сайт)
- Отдельный упрощённый layout (НЕ диспетчерский канбан): карточки «Мои заказы / Мои грузы /
  Склад / Документы», крупный трекинг-таймлайн. Та же дизайн-система (Golos Text), но без
  правого рейла «ИИ-координатор» — это внешний контур.
- AI-подсказки («рекомендуем дозаказать», «груз задержан») — снова **Итерация 1**, ghost-плейсхолдеры.

### Безопасность / разграничение (критично — внешний контур)
- **Жёсткая изоляция данных по `counterparty_id`** на уровне каждого запроса (row-level), не доверять
  фильтрам с фронта. Отдельная аутентификация контактов клиента (не staff-роли).
- Скачивание документов — только свои, через проверку владения + временные signed-URL.
- Никаких внутренних данных (себестоимость, маржа, резервы, чужие клиенты) в портал не отдавать.

### Межмодульные связи
- **portal → sales:** `portal.purchase_request.created` → sales создаёт лид/черновик заказа.
- **sales/logistics/wms → portal:** портал читает их витрины через `core`/события (read-models).
- Кандидат на **отдельный модуль `portal`** (а не внутрь `logistics`), т.к. агрегирует sales+wms+logistics
  и имеет свой внешний контур аутентификации. Решить при планировании.
